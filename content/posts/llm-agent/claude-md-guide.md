如果你正在使用 Claude Code，你可能已经跑通了安装、完成了第一个项目，甚至开始尝试多 Agent 并行。但真正拉开日常协作效率差距的，往往不是复杂的提示词技巧，而是一个只有几十行、却被称为 "AI 项目宪法" 的小文件：**CLAUDE.md**。

很多新手把它当技术文档写，结果上下文被塞满，AI 反而频频跑偏。今天这篇，带你彻底搞懂 CLAUDE.md 到底是什么、该怎么写、写什么，以及它和 Auto Memory 到底有什么区别。

## 🗺️ 一、CLAUDE.md 是什么？不是说明书，是"行为护栏"

CLAUDE.md 是 Claude Code 每次启动新会话时 **自动读取的第一个文件**。它不是 API 手册，也不是代码注释，而是你与 AI 之间关于"在这个项目里如何干活"的契约。

Anthropic 内部工程师和 Boris（Claude Code 创建者）团队反复强调一个定位：

> **它是护栏（Guardrails），不是手册（Manuals）。**

想象一下：Claude 是一个能力极强但刚入职的工程师。如果没有 CLAUDE.md，它就像空降进陌生代码库，什么都得从头猜；有了它，它一进来就知道项目的技术选型、自定义命令、历史坑点和团队红线。

Boris 团队管理 Claude Code 产品本身的核心规则文件，仅约 **2500 tokens（100行左右）**，却管住了整个产品的开发规范。文件短，不是功能弱，而是精准。

## 🛠️ 二、怎么写？从空文件开始的"纠错飞轮"

新手写 CLAUDE.md 最常踩的坑：**试图写一本百科全书**。把每个模块的作用、每个函数的签名全塞进去。结果 Claude 光读文件就吃掉大量上下文，真正写代码的空间被严重挤压。

### ✅ 正确姿势只有一个：从空文件开始，基于真实错误迭代。

这就是 Boris 提到的 **飞轮效应**。你不需要预先规划所有规则，而是：

- **第一周**：只写项目名、技术栈、启动/测试命令。
- **第二周**：Claude 跑错命令？装错包？用错状态库？立刻在文件里追加一条规则。
- **一个月后**：文件积累 20~30 条真实踩坑记录，Claude 的输出质量肉眼可见提升。
- **长期**：定期修剪过时规则，保持文件在 100行以内，并 **检入 Git 与团队共享**。

> 💡 **层级提示**：Claude Code 支持多级加载。个人通用偏好放 `~/.claude/CLAUDE.md`，项目规范放 `./CLAUDE.md`，Monorepo 可按模块放 `./src/CLAUDE.md`。按需放置，互不干扰。

## 📝 三、写什么？判断标准就一条

CLAUDE.md 的内容该多该少？记住这条黄金法则：

> **Claude 自己能从代码里读出来的，不写；猜不到的，必须写。**

### ✅ 必须写（AI 猜不到 / 容易错）

| 类型 | 示例 |
|------|------|
| 自定义命令 | `pnpm db:generate`、`npm run deploy:staging` |
| 非默认技术选型 | "状态管理用 Zustand，不要用 Redux" |
| 架构决策背景 | "为什么选 Server Component 而不是 useEffect 拉数据" |
| 环境陷阱 | "改环境变量必须重启 dev server"、"Drizzle 迁移后必须重新生成类型" |
| 明确禁令 + 替代方案 | "不要用 `--foo-bar` 标志，改用 `--baz`" |

### ❌ 绝对不写（浪费上下文）

| 类型 | 原因 |
|------|------|
| Claude 读代码就知道的事 | 如"这是一个 Next.js 项目""入口是 main.tsx" |
| 标准语言规范/废话 | 如"写整洁代码""遵循最佳实践""注意错误处理" |
| 详细 API 文档 | 给官方链接即可，全文粘贴会撑爆上下文 |
| 逐文件描述 | Claude 自带文件树扫描能力，不需要你手动列清单 |
| 频繁变动的信息 | 每次都要改的内容不适合固化在这里 |

### ⚠️ 三个避坑指南

1. **永远不要只写"禁止做 X"。** 当 Claude 觉得业务逻辑必须做 X 时，它会卡死或强行忽略。正确写法：`不要用 X，因为会导致 Y 问题，请改用 Z`。
2. **谨慎使用 @引用 大文件。** `@docs/xxx.md` 会在每次会话完整嵌入上下文。正确做法：写清路径+触发条件，如 `遇到 FooBarError 时，参阅 docs/troubleshooting.md`。
3. **命令太复杂？说明该简化了。** 如果某个 CLI 需要写三段话解释，不如封装成 bash wrapper，在 CLAUDE.md 里只记录简化后的命令。

## 🤖 四、CLAUDE.md vs Auto Memory：到底用哪个？

除了手写的 CLAUDE.md，Claude Code 还有一个 **自动记忆系统（Auto Memory）**。当你在对话中纠正 Claude（如"commit 以后用英文""测试文件放 `__tests__/`"），它会自动保存偏好到本地。

两者不是二选一，而是明确分工：

| 维度 | 手写 CLAUDE.md | Auto Memory (MEMORY.md) |
|------|---------------|------------------------|
| 维护方式 | 你主动编写/更新 | 对话纠正后 Claude 自动保存 |
| 作用域 | 项目级（检入 Git，团队共享） | 个人级（存在本地 `~/.claude/projects/...`） |
| 内容特点 | 结构化、精炼、面向团队规范 | 零散、按时间累积、面向个人习惯 |
| 典型场景 | 架构约束、自定义命令、常见陷阱 | "commit 用英文""用 2 空格缩进" |

> ✅ **最佳实践**：团队规范、项目架构、环境陷阱写进项目 CLAUDE.md；个人编码习惯、交互偏好交给 Auto Memory 自动处理。两者并行，互不冲突。

## 🚀 结语：让它自己"长"出来

CLAUDE.md 不是一次性写完就扔进仓库的静态文档，而是一个 **随着项目共同生长的活系统**。

很多高效用户的第一周，文件里只有 3 行；三个月后，它积累了 50 条精准规则，返工率断崖式下降。当你把这个文件维护好，你会发现：Claude 不再是那个需要你步步盯着的实习生，而是真正懂你项目历史的资深搭档。

**今天就能做的 3 步：**

1. 在项目根目录新建 `CLAUDE.md`，只写：项目名称、技术栈、启动/测试命令。
2. 正常开发，让 Claude 干活。
3. 只要它犯错或违背你的习惯，立刻追加一条规则。

别追求完美开局，追求持续迭代。三个月后回头再看，那个不到 100 行的文件，会是你最有价值的 AI 协作资产。
