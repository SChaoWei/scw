## 前言

AI编程工具在过去三年经历了三次范式跃迁：从VScode编辑器里面的Copilot的你写我猜代码补全，到Cursor的"编辑器内辅助开发"，再到Claude Code的"终端原生独立Agent"。它的出现我觉得在一定程度上互换了开发者与产品经理的之间的关系——以后往往程序员不再是坐在键盘前逐行敲代码的人，而是给AI下达指令、审查结果的产品负责人。

即便到了现在，仍然有不少开发者停留在网页端和 AI 逐条对话的阶段（前段时间给师弟做培训时，发现他也是这样，所以当即给他们进行培训，必须用我的中转站邀请码，哈哈哈）。同时很多初次接触 Claude Code 的人，也容易把它当成一个"会跑命令的聊天机器人"——边问边改、反复确认，Token 烧得快，上下文也越聊越乱。其实 Claude Code 内置了清晰的运行模式与一套高效的工作流，用对了方法，协作效率会有质的提升。

## 一、三种核心模式：让 Claude 按需"换挡"

Claude Code 不是只有一种工作状态。可以随时在以下模式间切换，让 AI 的行为匹配当前任务的风险与复杂度。

### 1. 默认交互模式（Interactive）

- **行为特征**：在每执行一步操作（创建文件、运行命令、Git 提交）都会弹出确认提示，需要手动批准，一个一个点。
- **适用场景**：初学者熟悉阶段、涉及生产环境/数据库/敏感配置的高风险操作、或希望严格掌控每一个中间步骤时。
- **注意**：长期使用容易产生"审批疲劳"——大量权限请求会被习惯性批准，安全机制的实际效果会打折扣。同时，人也遭不住这种重复性劳动。

### 2. Plan 模式（规划模式）

- **如何进入**：在提示符处切换到Plan模式（通过快捷键循环切换输入模式），Shift+Tab两次。
- **行为特征**：只规划，不执行。Claude 会读取代码库、分析依赖、输出详细的实现方案（包括修改哪些文件、采用什么技术栈、潜在风险等），但不会修改你的文件，也不会运行任何命令，非常类似于battle时候地拉通对齐，对齐一下颗粒度。
- **适用场景**：
  - 需求边界模糊，需要厘清实现路径
  - 涉及多文件重构或架构级调整
  - 刚接手新项目，不熟悉代码结构
  - 你需要和 AI "对齐认知"再动手
  - 进一步地总结来说，如果你需要跟同事解释才能让他做的任务，就值得用Plan模式。如果你能用一句话说清的任务，直接做就行。
### 3. Auto 模式（自动驾驶模式）

- **如何进入**：继续切换至 Auto 模式，或启动时使用自动权限参数。
- **行为特征**：内置风险分类器自动评估每个操作。安全操作（读文件、跑测试、装依赖）直接放行；危险操作（删远程分支、覆盖配置、访问凭证）才会拦截提示。
- **内部原理**：输入层采用个Prompt Injection探测器扫描读取的所有内容，如果内容看起来像在试图劫持Claude的行为，探测器会在内容传给Claude之前附加一条警告；输出层采用一个Transcript分类器会在每个操作执行前评估它的风险。分类器分两个阶段运行：第一阶段快筛，第二阶段细粒度推理（大部分安全操作在第一阶段就直接通过了，只有可疑操作才会触发第二阶段的深度分析。）
- **适用场景**：日常功能开发、重复性调试、已确认方案后的批量执行。
- **提醒**：操作生产数据库、云基础设施或敏感凭证时，因为Claude Code地上述措施存在漏检率原因，强烈建议切回手动确认。
- **其他**：还有一个dangerously-skip-permissions，这个大模型跳过所有权限提示且没有任何上述地防护措施，可能会删库跑路。
- **建议**：dangerously-skip-permissions和auto都不用，用/permissions来预授权某些命令，通过.claude/settings.json并提交到Git，让整个团队共享同一套权限配置。

## 二、黄金工作流：先 Plan → 沟通对齐 → 放手执行

很多人使用的时候觉得 Claude Code "容易跑偏"或"输出不稳定"，往往是因为跳过了plan阶段，直接让AI边想边做。更高效的方式是：**把纠结留在Plan阶段，执行阶段一气呵成**。

### Step 1：进入 Plan 模式，让 Claude 先出方案

不要一上来就写代码。先进入 Plan 模式，清晰描述需求：

> "我想为现有项目添加一个用户积分系统，支持签到、消费扣减、排行榜。先不要改代码，给我一个实现方案。"

Claude 会返回包含文件改动清单、技术选型、数据流设计的方案。此时你的角色是**技术评审**：

- "第三步换用 Redis 而不是内存缓存"
- "排行榜只需要 Top 10，不用分页"
- "参考 `src/auth/` 的现有结构，保持风格一致"

反复讨论，直到方案完全符合你的预期。Plan 模式的意义在于：**用自然语言完成架构评审，避免后续返工浪费 Token 和上下文**。

### Step 2：结构化沟通，固化执行指令

方案对齐后，将讨论结论整理成一份结构化的执行指令。你可以利用多行输入功能，将完整的技术要求一次性提交给 Claude：

```markdown
## 任务：用户积分系统
### 技术约束
- 使用 Redis 做排行榜缓存
- 数据库沿用现有 PostgreSQL
### 文件改动范围
- 新增 src/services/points.ts
- 修改 src/routes/user.ts
### 验收标准
- 所有现有测试通过
- 新增单元测试覆盖核心逻辑
```

这相当于给 AI 下发了一份技术 PRD，消除模糊空间。

### Step 3：切换执行模式，放手让 Claude 完成

确认指令无误后，切回正常模式或 Auto 模式。因为前期已经充分讨论，此时可以放心地让 Claude 一次性执行。

Claude 会自动完成 创建文件 → 安装依赖 → 编写逻辑 → 运行测试 → Git 提交 的全流程。你只需等待完成提示，检查输出是否符合预期。如有问题直接贴报错，Claude 会进入自动修复循环。

## 三、为什么这套流程能大幅提升效率？

### 避免"边做边改"的上下文污染

长对话中频繁纠正会挤占上下文窗口，导致 Claude "遗忘"早期约束。Plan 阶段集中解决歧义，执行阶段专注产出，上下文更干净。

### 从"监工"到"架构师"的角色跃迁

传统 AI 编程中，你盯着屏幕看它改每一行代码；这套流程中，你只负责定义边界、评审方案、验收结果。你的时间花在"写什么"和"对不对"，而不是"怎么写"。

### 节省 Token，降低试错成本

在错误基础上打补丁是最浪费 Token 的用法。先规划后执行，能显著降低整体 Token 消耗，且产出质量明显提升。

## 写在最后

Claude Code 的强大不在于它能自动跑命令，而在于它提供了一套完整的工程协作协议。Plan → 沟通 → Execute我觉得是一个非常标准的pipeline。

下次面对新需求时，不明白该怎么进行构建的时候，不妨先切到 Plan 模式进行探讨开会。你会发现，当你不再急于让 AI 敲下第一行代码，而是先把路想清楚时，剩下的事情，它真的能自己走完。

> **延伸建议**：
1、将每次 Claude 犯过的错沉淀到项目根目录的 `CLAUDE.md` 中。这个文件相当于 AI 的"项目记忆"，结合 Plan 工作流使用，会让 Claude 越来越懂你的代码库与团队规范。
2、/clear 的使用时机：/clear 会清空当前会话的所有对话历史，回到一个干净的起点。启动时读取的CLAUDE.md和项目文件不受影响。当你要开始一个和之前对话完全不同的任务时需要进行使用，此时如果不clear，Claude的上下文里还残留着大量关于那个API bug的信息，会干扰它对新任务的理解。
3、/compact和/btw：/compact在一个长会话当中，上下文太长影响了性能，用compact压缩一下上下文保留关键信息。/btw则是开启一个“侧链”对话，可以问Claude一个和当前任务不相关的问题，问完之后侧链结束，主对话的上下文不受影响。
4、一个会话啥都塞，非常容易让上下文被塞满，并且Claude对每个任务的理解都很浅。一个会话就聚焦于一个task，用完clear或者新开会话。
5、Claude跑偏之后进行纠正，纠正两次之后还是不行的话果断/clear重来。重新描述需求，说得更具体进行retry。
6、测试边界情况，写了一大堆代码进行测试，输出看着可以，没实际跑一下，容易出现边界错误，每一轮改动都建议跑一下。
7、项目需求要清晰明了，越具体，输出也就会越接近预期。
8、项目进行要写CLAUDE.md，不然每次新会话都要重新解释项目背景、代码规范、技术选型。
